本屋さんで覚えるGit入門

Pull・Commit・Push・Mergeの4ステップ完全ガイド

1登場人物と場所の整理
名前 本屋さんに例えると 役割
GitHub
（クラウド）		 本社 全データの「原本」を保管している金庫。
ローカル
（Mac）		 手元の作業机 本社からデータを借りてきて、実際に書き換える場所。
main
（ブランチ）		 売り場に並ぶ完成品 お客さんが見る本番。絶対汚してはいけない。
staging
（ブランチ）		 校正用のコピー 自由に書き込んでいいテスト用の本。
24つの魔法の言葉
プル (Pull) 本社の金庫から最新のデータを、自分の作業机に「持ってくる」。
コミット (Commit) 作業机で書いた内容に、「よし、これでOK！」とハンコを押す（まだ自分の机の上）。
プッシュ (Push) ハンコを押したデータを、本社の金庫（GitHub）に「送る」。
マージ (Merge) 本社の金庫の中で、校正用の本（staging）の内容を、完成品（main）に「合体させる」。
3修正依頼〜完了までの「100%成功マニュアル」
STEP 1作業机を最新にする（出社）
まず、本社の金庫にある最新版を、自分の机に持ってきます。
「main ブランチに切り替えて、最新の状態を pull してきて。そのあと staging に切り替えて、main の内容を反映させて（同期して）。」
STEP 2AIに修正させる（作業）
ここで初めて、中身を書き換えます。
「index.html の見出しを『〇〇』に変えて。」
✅ 自分の確認：Finderでファイルを開いて、ブラウザで見た目が変わったかチェック！
STEP 3金庫に送る（出荷）
机でできた内容を、GitHubの金庫（staging）に送ります。
「修正内容を commit して、GitHub に push して。」
※これでGitHubの「stagingブランチ」が更新されます。
STEP 4本番に合体させる（開店）
最後はブラウザ（GitHubの画面）で、本番の完成品に反映させます。
GitHubでの操作：Pull Request を作る → 中身をチェックして、Merge ボタンを押す（合体完了！）。
💡 なぜ「ローカル」と「GitHub」に分かれているの？
もしGitHub（金庫）を直接いじって、書き換え中にネットが切れたり、間違えて全部消しちゃったりしたら、取り返しがつきません。
だから、「一度手元（ローカル）に持ってきて、納得いくまで直してから、最後にまとめて金庫（GitHub）に送る」という仕組みになっているんです。
